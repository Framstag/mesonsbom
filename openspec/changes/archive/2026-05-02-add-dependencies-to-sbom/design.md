## Context

The current `mesonsbom` tool generates a CycloneDX SBOM containing only the main application component.  Meson’s introspection output includes `intro-dependencies.json`, which lists each direct library dependency with its name and version.  To produce a useful SBOM for supply‑chain security, we need to incorporate these libraries as components and represent the dependency relationships.

## Goals / Non‑Goals

**Goals:**
- Parse `intro‑dependencies.json` from the Meson build directory supplied via `--build-dir`.
- For each dependency that is actually resolved in the current build (skip optional dependencies), add a CycloneDX component of type **library** with the correct `name` and `version`.
- Record a `dependsOn` edge from the main application component to each library component.
- Keep the command‑line interface unchanged.
- Extend the integration test to verify that a `library` component appears in the generated SBOM.

**Non‑Goals:**
- Resolve transitive dependencies beyond those listed in `intro‑dependencies.json`.
- Add licensing, purl, or additional metadata for the libraries.
- Support other build systems (e.g., CMake, Bazel) in this iteration.

## Decisions

1. **Input file** – Use `intro‑dependencies.json` (generated by `meson introspect`).  *Rationale:* Provides a reliable list of direct dependencies with version information.
2. **Component type** – Model each library as a CycloneDX component with `"type": "library"`.  *Rationale:* Aligns with CycloneDX’s taxonomy for linked third‑party code.
3. **Dependency edge** – For every library component, create a `dependsOn` entry referencing the main application component (`projName`).  *Rationale:* Explicitly expresses the direct linkage in the SBOM graph.
4. **Optional dependencies** – Skip entries where `"type"` is `"optional"` to ensure only resolved dependencies are added.  *Rationale:* Prevents inclusion of libraries not present in the build.
5. **Error handling** – Throw a clear `std::runtime_error` if `intro‑dependencies.json` cannot be opened.  *Rationale:* Immediate feedback for mis‑configured build directories.
6. **Testing** – Extend `tests/integration_test.cpp` to assert that the generated SBOM contains at least one component with `"type": "library"`.  *Rationale:* Guarantees the new extraction logic is exercised and prevents regressions.

## Risks / Trade‑offs

- **Future JSON format changes** – If Meson alters the structure of `intro‑dependencies.json`, parsing may miss data.  *Mitigation:* The code checks that the parsed JSON is an array and safely skips entries lacking `name` or `version`. A clear runtime error is raised if the file cannot be read.
- **Version accuracy** – The version reported by Meson may not match the exact runtime library version (e.g., system‑installed variants).  *Mitigation:* Document this limitation; for most use‑cases the Meson‑reported version is sufficient.
- **Performance impact** – Adding an extra JSON parse adds negligible overhead for typical project sizes.  *Mitigation:* No further action required.

## Open Questions

- Dependencies will only be added to the SBOM when they are actually present in the build (i.e., resolved). No additional switch is required to control this behavior.
- The design therefore does not include a command‑line flag for disabling dependency extraction; the tool automatically skips optional dependencies that are not resolved.
